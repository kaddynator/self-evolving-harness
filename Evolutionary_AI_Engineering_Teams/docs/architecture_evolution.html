What this system is

A compiler and optimizer for AI agent organizations. You describe a task in plain language; the system synthesizes a multi-agent workflow, runs it, measures it, finds why it underperformed, and rewrites the workflow itself — generation after generation. The artifact it returns is not just the task result, it is the optimized organization: which agents exist, what their prompts say, which tools and models they get, how they hand off, and how big their budgets are.

Compile

Gemini turns a natural-language task into a validated Harness IR — agents, prompts, tools, phases, communication edges, evaluation contract.

Run & Evaluate

Agents execute in sandboxed phases, calling tools via Gemini function-calling. A weighted-sum scorer plus an LLM judge grade the run.

Mine & Mutate

A miner extracts failure signatures; the engine maps each failure mechanism to a concrete workflow mutation.

Gate & Chain

A validation gate accepts a candidate only on no regression + a real improvement. The winner becomes the parent of the next generation.

End-to-end architecture

A single closed loop. The natural-language task enters once at the left; everything to the right of the compiler repeats every generation, with the accepted candidate feeding back into the next cycle.

Anatomy of one generation

EvolutionPipeline.run_cycle() executes nine stages in order. The first run of a generation is the parent; each proposed mutation produces a candidate that is run for real and compared against the parent before the gate decides.

How a workflow evolves across generations

Each accepted mutation changes one surface of the organization — a prompt, a budget, a tool list, a model, or the agent roster — and the score climbs as redundancy is pruned and weak agents are strengthened. The example below traces a bloated 7-agent workflow being optimized toward a tight, high-scoring core.

Mutation types

Every failure mechanism the miner detects is mapped to one or more concrete edits. Only mutations listed in the harness's mutation_policy.allowed_mutations are generated, and protected components (success conditions, the validation gate itself) are never touched.

modify_prompt — grow/specialize an agent's prompt (e.g. tie every acceptance criterion to a test assertion, or enforce a minimal-diff methodology). Uses Gemini to genuinely expand the prompt when a client is available, else a deterministic append.
adjust_budget — tighten an agent's max_tool_calls to curb runaway exploration and drive tool_calls down.
modify_tools — grant a tool that was denied at runtime (resolves wrong_tool_permission failures).
change_model — upgrade an agent to a stronger model (e.g. requirements agent → gemini-2.5-flash) for more precise output.
remove_agent — prune a non-core agent flagged as redundant, lowering overhead and improving the efficiency score. Core agents (requirements/coder/tester/reviewer) and sole-in-phase agents are protected.
add_agent — insert a dedicated verifier when a run concluded without test evidence (unverified_completion).
modify_runtime_policy — enforce artifact-before-finish, prevent identical retries, or cap repeated tool errors.
reorder_edges — make a hand-off blocking so the tester is guaranteed to run after the coder (fixes late_testing).

The validation gate

A candidate is only allowed to replace its parent when it earns it. The gate applies three rules to the candidate's real evaluation versus the parent's:

  • runtime The candidate must finish within max_runtime_seconds, else rejected outright.
  • no regression No metric in require_no_regression (e.g. tests_pass) may drop below the parent.
  • improvement At least one metric in require_improvement_any (e.g. total_score or lower tool_calls) must strictly improve.

On the very first run there is no parent to compare against, so the gate accepts simply on passing the success threshold. This accept-only-on-net-improvement rule is what makes evolution monotonic: scores never go backward, and the optimizer keeps the best organization it has found.

Throughout the loop, every stage publishes events to the observability bus (evolution_start, generation_start, evaluation_complete, weakness_mined, mutation_proposed, gate_decision, harness_snapshot, evolution_complete) so this dashboard can render the workflow topology, scores, and decisions live as they happen.
