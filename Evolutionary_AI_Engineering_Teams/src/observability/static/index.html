Architecture & Evolution

What this system is

A compiler and optimizer for AI agent organizations. You describe a task in plain language; the system synthesizes a multi-agent workflow, runs it, measures it, finds why it underperformed, and rewrites the workflow itself — generation after generation. The artifact it returns is not just the task result, it is the optimized organization: which agents exist, what their prompts say, which tools and models they get, how they hand off, and how big their budgets are.

Compile

An LLM compiler (Claude or Gemini) turns a natural-language task into a validated Harness IR — agents, prompts, tools, phases, communication edges, evaluation contract.

Run & Evaluate

Agents execute in sandboxed phases, calling tools via the model's function-calling. A weighted-sum scorer plus a real LLM-as-judge (Claude Sonnet 4.6 / Gemini 3.5 Flash), on by default, grade the run.

Mine & Mutate

A miner extracts failure signatures; the engine maps each failure mechanism to a concrete workflow mutation.

Gate & Chain

A validation gate accepts a candidate only on no regression + a real improvement. The winner becomes the parent of the next generation.

Feedback Flywheel & Eval Dataset

Evolution is only as good as the signal it optimizes against. The system runs in two evaluation modes and closes a human-in-the-loop flywheel so the dataset — and therefore the organization — keeps getting better in production.

Mode A · Ground-truth eval cases

When labeled data exists, runs are graded reference-based: each EvalCase carries a customer input and a human-authored expected_output, and the agent's actual output is scored against it.

Mode B · LLM-as-judge

When no labels exist, a real LLM-as-judge grades the run. It is on by default — no longer optional. Backends: Claude Sonnet 4.6 via Vertex AI (preferred, ADC auth) with Gemini 3.5 Flash via the Generative Language API key as fallback. The legacy Vertex-Gemini publisher client is deprecated (it 404s).

The flywheel loop

A deployed workflow doesn't just answer customers — it mines its own failures and feeds them back as new training data. The loop runs continuously:

  • Sentinel is platform-level. The feedback/sentiment agent is injected by the platform and exempt from evolution, so the optimizer can never prune or rewrite the very component that captures training signal.
  • Capture on negative sentiment. When the sentinel flags a bad answer, it writes an EvalCase with status=needs_label, freezing the input, a context_snapshot, and the agent's actual_output.
  • Human supplies the label. An admin (or the end user) opens the Eval Dataset tab and writes the expected_output; the case flips to status=labeled.
  • Only labeled cases drive evolution. The trigger fires on a batch / threshold of newly labeled cases — never on raw unlabeled ones.
  • Gate against the full dataset. Before any redeploy, the evolved agent is graded reference-based against the entire labeled dataset to guarantee no regression. (The scheduler that orchestrates this batching/triggering is the orchestration layer and is not yet fully built.)

Three per-agent optimization knobs

Evolution now tunes three surfaces of every agent, not two: model prompt tools — the model it runs on, the instructions it follows, and the tool surface it can call.

The eval_cases collection

Captured cases live in the eval_cases MongoDB collection. Each EvalCase record carries:

  • id · agent_id — which deployed agent produced the answer.
  • input — the customer question.
  • context_snapshot — frozen state the agent saw (so the case is reproducible).
  • actual_output — what the agent actually returned.
  • feedback · sentiment — the sentinel's verdict.
  • expected_output — the human-authored ground truth (filled at label time).
  • status — needs_label → labeled; labeled_by · created_at.
Worked example · FinOps

input: "why did my GCP bill spike + how to cut it"

context_snapshot (frozen): billing total = $48,200 this month vs $31,400 last month; 14 idle Compute Engine instances; 3 un-attached SSD persistent disks; a Cloud SQL HA pair at 7% CPU.

expected_output (structured, ranked by $-savings):

  • 1 · Delete / right-size 14 idle instances → ~$9.1k/mo.
  • 2 · Downgrade the over-provisioned Cloud SQL HA pair → ~$3.4k/mo.
  • 3 · Remove 3 orphaned SSD disks + stale snapshots → ~$1.2k/mo.
  • 4 · Apply committed-use discounts on the steady-state baseline → ~$2.0k/mo.

The agent's answer is then graded reference-based against this ranked, dollar-quantified expected_output — not a vibe check.

End-to-end architecture

A single closed loop. The natural-language task enters once at the left; everything to the right of the compiler repeats every generation, with the accepted candidate feeding back into the next cycle.

Anatomy of one generation

EvolutionPipeline.run_cycle() executes nine stages in order. The first run of a generation is the parent; each proposed mutation produces a candidate that is run for real and compared against the parent before the gate decides.

How a workflow evolves across generations

Each accepted mutation changes one surface of the organization — a prompt, a budget, a tool list, a model, or the agent roster — and the score climbs as redundancy is pruned and weak agents are strengthened. The example below traces a bloated 7-agent workflow being optimized toward a tight, high-scoring core.

Mutation types

Every failure mechanism the miner detects is mapped to one or more concrete edits. Only mutations listed in the harness's mutation_policy.allowed_mutations are generated, and protected components (success conditions, the validation gate itself) are never touched.

modify_prompt — grow/specialize an agent's prompt (e.g. tie every acceptance criterion to a test assertion, or enforce a minimal-diff methodology). Uses the LLM (Claude or Gemini) to genuinely expand the prompt when a client is available, else a deterministic append.
adjust_budget — tighten an agent's max_tool_calls to curb runaway exploration and drive tool_calls down.
modify_tools — grant a tool that was denied at runtime (resolves wrong_tool_permission failures).
change_model — upgrade an agent to a stronger model (e.g. requirements agent → gemini-2.5-flash) for more precise output.
remove_agent — prune a non-core agent flagged as redundant, lowering overhead and improving the efficiency score. Core agents (requirements/coder/tester/reviewer) and sole-in-phase agents are protected.
add_agent — insert a dedicated verifier when a run concluded without test evidence (unverified_completion).
modify_runtime_policy — enforce artifact-before-finish, prevent identical retries, or cap repeated tool errors.
reorder_edges — make a hand-off blocking so the tester is guaranteed to run after the coder (fixes late_testing).

The validation gate

A candidate is only allowed to replace its parent when it earns it. The gate applies three rules to the candidate's real evaluation versus the parent's:

  • runtime The candidate must finish within max_runtime_seconds, else rejected outright.
  • no regression No metric in require_no_regression (e.g. tests_pass) may drop below the parent.
  • improvement At least one metric in require_improvement_any (e.g. total_score or lower tool_calls) must strictly improve.

On the very first run there is no parent to compare against, so the gate accepts simply on passing the success threshold. This accept-only-on-net-improvement rule is what makes evolution monotonic: scores never go backward, and the optimizer keeps the best organization it has found.

Throughout the loop, every stage publishes events to the observability bus (evolution_start, generation_start, evaluation_complete, weakness_mined, mutation_proposed, gate_decision, harness_snapshot, evolution_complete) so this dashboard can render the workflow topology, scores, and decisions live as they happen.

What is this?

The Harness Evolution Monitor is an agent builder and optimizer. You give it a plain-language task description; it synthesizes a multi-agent workflow (an Organization Harness), runs it, evaluates it, mines failure patterns, and proposes mutations — repeating for N generations until no further improvement is accepted.

The output is the optimized workflow definition (the harness YAML), not just the task result. Think of it as a compiler + optimizer for AI organizations.

Domains — how they differ

The domain you pick decides the starting workflow blueprint: which agents are created, the phase sequence they run in, and which artifacts/metrics the evaluator expects. Evolution then optimizes from that blueprint.

⚙️

Software Engineering

requirements → coder → tester → reviewer

A build-test-review loop. Scored on a passing test suite and an explicit code review, penalized for excess tool calls.

Artifacts: code_patch, test_results

Best for: features, bug fixes, refactors, APIs

🔬

Research

researcher → analyst → writer

A gather-analyze-synthesize flow that produces a written report rather than code. No test/review gate.

Artifacts: research_notes, final_report

Best for: literature reviews, comparisons, summaries

🗄️

Data Pipeline

fetcher → processor → reporter

An ETL flow: ingest raw data, transform it, and emit a summary. Optimized for clean, staged data handoffs.

Artifacts: raw_data, processed_data, summary_report

Best for: ETL, scraping, ingestion, transformation

🧩

General

planner → executor → verifier

A domain-agnostic plan-do-check loop. The safe default when the task doesn't fit the others.

Artifacts: output

Best for: anything that isn't clearly one of the above

Evolution Pipeline — 9 Stages

1

Compile — Convert natural language task → Organization Harness IR (YAML with agents, prompts, tools, graph, evaluation policy).

2

Run — Execute the harness phase by phase. Each agent receives shared memory from prior agents and emits artifacts.

3

Evaluate — Score the run: weighted sum of binary checks (tests pass, feature works, reviewer accepts) plus penalties (tool calls, diff size). Threshold: 70.

4

Mine weaknesses — 8 rule-based classifiers scan the execution trace to produce FailureSignature structs (mechanism + agent behavior + cause).

5

Propose mutations — Each failure mechanism maps to one or more mutation operators (modify_prompt, modify_tools, adjust_budget, add_agent, remove_agent, reorder_edges…).

6

Run candidates — Each mutated harness is fully executed (same executor, same mock or real LLM backend — Claude or Gemini).

7

Validation gate — A candidate is accepted only if: no regression on protected metrics AND improvement on at least one target metric.

8

Persist — All runs, evaluations, mutations, and lessons are stored in MongoDB (or mongomock for local demo). Lessons feed back into future compilation prompts.

9

Chain — If a candidate was accepted, it becomes the next generation's base. Repeat up to max_generations.

Organization Harness IR

The IR is a YAML file with 11 top-level sections:

organization:   id, name, version, objective, domain, constraints
task:           id, title, description, inputs, success_conditions, artifacts_expected
agents:         id, name, role, prompt, tools[], budget, memory_policy, output_contract
communication:  topology, edges[], shared_memory
execution:      mode, phases[], retry_policy, stopping_conditions
runtime_policies: max_tool_calls, require_artifact_before_finish, …
failure_recovery: on_tool_error, on_missing_artifact, …
evaluation:     metrics[], binary_checks[], scoring, validation_gate
weakness_mining: enabled, failure_signature_fields
mutation_policy: allowed_mutations[], protected_components[], proposal_width
observability:  trace_level, log_events[]

Failure Mechanisms

  • weak_requirements_grounding — Requirements agent produced vague criteria → tests fail
  • missing_required_artifact — An agent finished without producing its expected artifact
  • wrong_tool_permission — Agent called a tool not in its allowed list
  • tool_overuse — Agent exceeded the exploration-to-implementation threshold
  • unverified_completion — Run concluded without test evidence
  • review_without_diff — Reviewer ran before coder produced a diff
  • late_testing — Coder ran tests before the dedicated tester phase
  • timeout_risk — Run elapsed time approached the runtime limit
